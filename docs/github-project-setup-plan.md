# GitHub Project 建立方案

> 适用范围：当前 `AI 学习助手` 项目，单仓库、单人开发、以 v1 产品落地为主。
>
> 相关文档：
> - [product-prd.md](./product-prd.md)

## 1. 文档目的

这份文档不是为了设计一套“标准团队流程”，而是为了给当前项目建立一套足够轻、足够稳、能长期维护的 GitHub Project 方案。

结合 PRD，当前项目的核心目标非常明确：

- 把产品从“通用聊天工具”收敛为“学习型 AI 产品”
- 围绕内容阅读建立核心体验，而不是围绕输入框堆功能
- v1 聚焦一条主路径：
  - 选中一段 AI 回复内容
  - 发起局部追问
  - 通过轻量弹出对话框完成追问
  - 关闭后在原文留下标记
  - 后续再次点击标记重新进入追问
  - 支持解除追问链接

因此，Project 的作用不应该是“把工作管理得很复杂”，而应该是：

- 帮助我们始终聚焦 v1 主路径
- 让 backlog、当前任务和已完成内容清晰分离
- 用最低维护成本支撑持续开发

---

## 2. 建立原则

### 原则一：Project 服务产品收敛，不服务流程表演

当前阶段最重要的不是字段齐全，也不是视图丰富，而是确保所有卡片都在为 PRD 中的核心体验服务。

如果一个字段、一个标签、一个流程步骤，不能帮助我们更快判断“这项工作是否推进了原位追问体验”，就不应该进入第一版配置。

### 原则二：单人开发优先减少维护动作

当前只有一个仓库、一个开发者，GitHub Project 的设计应优先减少重复操作：

- 少字段
- 少视图
- 少状态
- 少手工同步

### 原则三：底层简单，表达明确

Project 只需要清楚回答四个问题：

- 这件事是什么类型的工作
- 它是否属于 v1 核心路径
- 它现在处于什么状态
- 它是不是当前最该做的事情

---

## 3. 推荐建立方式

建议创建 **1 个个人 Project**，作为当前仓库的唯一产品执行盘。

推荐名称：

- `AI 学习助手 v1`

推荐描述：

- `围绕原位追问体验推进 v1：选中追问 -> 轻量弹框 -> 原文标记 -> 再次进入 -> 解除链接`

这个 Project 只承载三类 item：

- `Issue`
- `Pull Request`
- `Draft issue`

其中：

- `Issue` 用于正式任务
- `Pull Request` 自动关联，帮助确认交付闭环
- `Draft issue` 用于暂存想法，不打断当前工作

不建议当前阶段再拆多个 Project。单仓库、单人开发下，多盘会带来不必要的切换成本。

---

## 4. 推荐字段

当前建议只保留 **4 个字段**。

### 4.1 状态

类型：`Single select`

建议值：

- `待整理`
- `下一步`
- `进行中`
- `已阻塞`
- `已完成`

说明：

- `待整理`：先记录，暂不处理
- `下一步`：下一批最值得推进的事项
- `进行中`：当前正在做
- `已阻塞`：受外部条件或技术判断卡住
- `已完成`：已完成

其中 `下一步` 很重要。对于单人开发，它比 `Todo` 更有用，因为它天然表示“接下来要做什么”，有助于避免 backlog 过大时失焦。

### 4.2 优先级

类型：`Single select`

建议值：

- `P0`
- `P1`
- `P2`

说明：

- `P0`：直接阻塞 v1 主路径
- `P1`：重要，但不一定先于所有事项
- `P2`：可延后优化或次级改进

### 4.3 类型

类型：`Single select`

建议值：

- `阶段目标`
- `功能`
- `缺陷`
- `重构`
- `文档`

说明：

- `阶段目标`：阶段性目标或较大工作包
- `功能`：新增产品能力
- `缺陷`：问题修复
- `重构`：结构调整、弱化旧能力、清理技术债
- `文档`：产品、测试、交付相关文档

### 4.4 阶段

类型：`Single select`

建议值：

- `P0 范围收敛`
- `P1 基础底座`
- `P2 原位追问`
- `P3 标记与回到原文`
- `P4 上线准备`

说明：

这个字段直接对应现有路线图，用于把产品目标和执行阶段对齐。

- `P0 范围收敛`：收敛范围与切换方向
- `P1 基础底座`：数据结构、API、prompt、runtime 底座
- `P2 原位追问`：选区检测与轻量追问框主路径
- `P3 标记与回到原文`：标记、再次进入、解除链接
- `P4 上线准备`：测试、回归、上线准备

### 不建议现在加入的字段

以下字段暂时不建议加入：

- `Size`
- `Risk`
- `Iteration`
- `Sprint`
- `Estimate`
- `Depends On`
- `Area`

原因很简单：它们在多人协作时更有价值，但当前阶段大概率会增加维护负担，收益不足。

## 5. 推荐视图

当前建议只建立 **3 个主要视图**。

### 5.1 路线图

布局：`Table`

用途：

- 作为总览视图
- 按 `阶段` 分组
- 按 `优先级` 排序
- 默认过滤掉 `已完成`

这个视图回答的问题是：

- v1 当前有哪些工作
- 这些工作分别处于哪个阶段
- 现在最应该推进的是哪些卡片

建议展示列：

- `标题`
- `状态`
- `优先级`
- `类型`
- `阶段`

### 5.2 当前聚焦

布局：`Board`

按 `状态` 分列：

- `待整理`
- `下一步`
- `进行中`
- `已阻塞`
- `已完成`

用途：

- 每天打开后的主工作面板
- 帮助控制并发任务数量

使用建议：

- `进行中` 同时最多保留 1 到 2 项
- `下一步` 保持在 3 到 5 项以内

### 5.3 完成记录

布局：`Table`

过滤：

- `状态 = 已完成`

用途：

- 回顾近期完成内容
- 为发布说明、开发日志、阶段复盘提供素材

单人项目很容易出现“做了很多，但感受不到进展”的情况。保留一个清晰的完成记录视图，会很有帮助。

### 可选视图：待整理

如果后续 `Draft issue` 明显增多，可以补一个 `Inbox` 视图：

- 布局：`Table`
- 过滤：`状态 = 待整理`

当前不是必须项。

## 6. Labels 使用建议

Project fields 负责表达工作流，labels 只负责表达技术属性。

建议保留少量 labels，例如：

- `frontend`
- `api`
- `ai`
- `data`
- `test`
- `ux`

不建议把以下信息同时放在 labels 和字段里：

- 状态
- 优先级
- 阶段
- 工作类型

也就是说：

- `优先级` 用字段，不用 `P0/P1/P2` labels
- `类型` 用字段，不用 `feature/bug/docs` labels
- `阶段` 用字段，不用 `v1/v2/m1/m2` labels

这样可以避免同一件事维护两遍。

## 7. 推荐的阶段目标结构

为了让 Project 与 PRD 对齐，建议先建立 5 个阶段目标，直接对应产品目标的推进顺序。

### 阶段目标 1：产品范围收敛与切换

- `类型`: `阶段目标`
- `Priority`: `P0`
- `阶段`: `P0 范围收敛`

目标：

- 明确 v1 不再继续强化“通用聊天工具”叙事
- 明确 Artifact 工作台在 v1 中被弱化
- 明确 v1 只聚焦“原位追问”的主体验

### 阶段目标 2：原位追问基础底座

- `类型`: `阶段目标`
- `Priority`: `P0`
- `阶段`: `P1 基础底座`

目标：

- 建立追问子对话的数据模型
- 明确锚点、追问消息、重新进入和解除链接的生命周期

### 阶段目标 3：原位追问主路径

- `类型`: `阶段目标`
- `Priority`: `P0`
- `阶段`: `P2 原位追问`

目标：

- 跑通“选中内容 -> 发起追问 -> 弹出轻量对话框 -> 连续追问”的主路径

### 阶段目标 4：标记、再次进入与解除链接

- `类型`: `阶段目标`
- `Priority`: `P0`
- `阶段`: `P3 标记与回到原文`

目标：

- 让追问与原文形成闭环
- 支持点击标记重新进入
- 支持解除链接

### 阶段目标 5：测试验证与上线准备

- `类型`: `阶段目标`
- `Priority`: `P1`
- `阶段`: `P4 上线准备`

目标：

- 保证核心路径可回归、可上线、可演示

## 9. 推荐的日常使用方式

### 每天开始时

- 打开 `当前聚焦`
- 从 `下一步` 中挑 1 个最重要的卡片移到 `进行中`

### 有新想法时

- 先建成 `Draft issue` 或普通 `Issue`
- 默认放进 `待整理`
- 不因为新想法打断当前 `进行中`

### 一个任务完成时

- 关闭对应 issue 或把状态改为 `已完成`
- 如果已经有 PR，确保 PR 自动关联

### 每周做一次整理

- 清理 `待整理`
- 重新选出 3 到 5 个 `下一步`
- 删除或关闭已经不符合产品方向的旧卡片

这个节奏已经足够支撑当前项目，不需要再引入 sprint、story point 或复杂评审流程。

---

## 10. 自动化建议

建议只启用最少量自动化：

- 新建的 issue 和 PR 自动加入 Project
- PR merge 或 issue close 后自动设为 `已完成`
- `已完成` 项在一段时间后自动归档

当前阶段不建议引入更复杂的自动化规则。对于单人项目，过度自动化往往会比手动维护更难理解。

---

## 11. 与产品文档的关系

本方案直接服务 PRD 中的核心目标：

- 学习导向高于通用聊天
- 原位追问高于全局重问
- 不打断阅读高于功能堆叠

---

## 12. 最终建议

对于当前项目，最合适的 GitHub Project 不是“完整管理系统”，而是一个轻量的产品执行面板。

建议最终配置收敛为：

- 1 个 Project
- 4 个字段：`状态`、`优先级`、`类型`、`阶段`
- 3 个主要视图：`路线图`、`当前聚焦`、`完成记录`
- 少量 labels，仅表达技术属性
- 5 个阶段目标，对应 v1 核心推进阶段

这样既能对齐产品目标，也不会让维护成本反过来拖慢开发。
