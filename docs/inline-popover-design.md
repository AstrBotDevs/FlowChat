## 一、设计概述

### 方案选择

**Inline Popover（内联气泡）**——用户在 AI 回复中选中文字后，在选区正下方弹出一个紧凑的追问气泡。气泡内包含输入框和对话区域，视觉上像"在书上做批注"。

### 核心设计原则

| 原则   | 含义                                 |
| ------ | ------------------------------------ |
| 足够轻 | 不像新开一个聊天页面，视觉和认知负担最小 |
| 足够近 | 出现位置紧贴用户刚刚选中的内容         |
| 足够稳 | 不会误伤当前阅读状态，不推移主聊天布局   |
| 足够连续 | 关闭后可回到原文，锚点可再次进入        |

### 适用范围

- 仅 assistant 角色的消息文本可被选中追问
- 用户消息、系统 UI、工具调用结果不触发追问交互

## 二、选区触发机制

### 交互流程

```
用户拖选文字 → mouseup + 短延迟 → 浮现"追问"按钮 → 点击 → 展开 Popover
```

### Step 1：选中文字

用户在 AI 回复的文本内容中通过鼠标拖选（或触屏长按选择）一段文字。选区以半透明背景高亮。

### Step 2：浮现追问按钮

选区确定后（mouseup 事件 + 短延迟防误触），在选区正下方居中浮现一个迷你追问按钮，带对话气泡图标和「追问」文字。

### Step 3：点击"追问"展开 Popover

点击追问按钮后：

1. 按钮消失
2. 选区文字保持高亮，底部增加品牌色下划线
3. Popover 在选区正下方展开，带入场动画
4. 输入框自动获得焦点

### 触发规则

| 规则         | 说明                                                |
| ------------ | --------------------------------------------------- |
| 触发时机     | mouseup + 短延迟，防止拖选过程中误触                  |
| 按钮消失     | 点击空白处、选区变化、或滚动页面时自动消失              |
| 选区范围限制 | 仅 assistant 消息中的纯文本内容，代码块内文字不触发     |

## 三、Popover 内部结构

### 整体布局

Popover 由三部分组成，从上到下：

```
┌─────────────────────────────────┐
│  Header：引用标识 + 关闭按钮     │
├─────────────────────────────────┤
│  Conversation：对话内容区域       │
│  （初始态隐藏，有消息后出现）      │
├─────────────────────────────────┤
│  Input：输入框 + 发送按钮         │
└─────────────────────────────────┘
```

### 定位策略

- **优先**：选区正下方，左对齐于选区起始位置
- **翻转**：选区距视口底部不足 Popover 高度时，翻转到选区上方
- **水平修正**：Popover 右侧超出视口时向左偏移，确保完整可见
- **滚动跟随**：页面滚动时 Popover 跟随选区移动（position 相对于消息容器）

### Header 区域

左侧显示图标 + "追问"标签 + 引用文字（单行截断）。右侧显示关闭按钮。多轮对话时增加轮次计数徽章。

### 三种状态

#### 状态 A：初始态（等待输入）

- Header 显示引用文字
- 对话区域隐藏（无内容）
- 输入框显示 placeholder
- 发送按钮置灰

#### 状态 B：回复中（流式输出）

- 对话区域出现，显示用户消息和 AI 流式回复
- AI 回复末尾显示闪烁光标
- 输入框禁用
- 发送按钮变为停止按钮

#### 状态 C：多轮对话

- 对话区域显示所有历史消息
- **历史折叠**：超过 2 轮时，之前的 AI 回复自动折叠为单行摘要，点击可展开
- 最新一轮 AI 回复始终完整展示
- Header 显示轮次计数徽章

### 消息样式

- **用户消息**：右对齐，带浅色背景气泡
- **AI 消息**：左对齐，带小图标，无背景色，支持 Markdown 渲染
- **折叠态 AI 消息**：左侧竖线 + 单行摘要文字 + 「展开」链接

### 关闭方式

| 方式              | 行为                          |
| ----------------- | ----------------------------- |
| 点击 × 按钮       | 关闭 Popover，保存锚点        |
| 按 Esc 键         | 关闭 Popover，保存锚点        |
| 点击 Popover 外部  | 关闭 Popover，保存锚点        |
| 打开另一个追问     | 当前 Popover 自动关闭，保存锚点 |

**同一时刻只能打开一个 Popover。**

## 四、认知锚点

### 锚点的产生

当用户关闭一个**至少发送过一条消息**的追问 Popover 时，系统自动在原文中创建认知锚点。如果用户打开 Popover 但未发送任何消息就关闭，则不创建锚点。

### 锚点的视觉表现

被追问过的文字段落获得以下视觉标记：

- **下划线**：品牌色虚线下划线，不改变文字颜色
- **数字徽章**：上标小数字，显示该处追问的对话轮数
- **Hover 态**：下划线变实线，鼠标变为 pointer，出现 tooltip 预览第一轮 Q&A 摘要

### 锚点的视觉优先级

锚点标记必须极其克制，不能让正文变得花哨：

- 默认态几乎不影响阅读——仅淡色虚线
- 只有 hover 时才加强视觉反馈
- 锚点过密时不应造成视觉干扰

### 锚点的再次进入

点击带锚点标记的文字，重新打开对应的追问 Popover，加载历史对话内容。用户可以继续追问或仅回顾之前的对话。

## 五、嵌套追问

### 交互流程

在追问 Popover 内的 AI 回复中，用户同样可以选中文字发起二次追问。这触发一个新的嵌套层级：

```
主消息流
  └── AI 回复中选中"梯度下降"  → Popover Level 1
        └── AI 回复中选中"学习率"  → Popover Level 2
              └── 继续嵌套...
```

### 嵌套的视觉处理

**不使用 Popover 套 Popover 的方式。** 嵌套追问采用"替换 + 面包屑导航"模式：

1. 用户在 Level 1 Popover 中选中文字并追问
2. 当前 Popover 内容平滑过渡为 Level 2 的对话
3. Header 区域增加面包屑路径，显示追问层级

```
┌─────────────────────────────────────────────────┐
│  ← [「监督学习...」] › [「学习率」]        [×]    │
├─────────────────────────────────────────────────┤
│  （Level 2 的对话内容）                           │
├─────────────────────────────────────────────────┤
│  [输入框]                               [发送]   │
└─────────────────────────────────────────────────┘
```

- 点击面包屑中的上层标题可回到上一层对话
- `←` 按钮等同于回到上一层
- 层级之间的切换使用左右滑动过渡动画

### 嵌套层级

- 无硬性限制（与 core-logic.md 一致）
- 面包屑超过 3 层时折叠为 `... › [当前层]`，点击 `...` 展开完整路径

## 六、锚点管理

### 解除追问链接

用户可以移除某个认知锚点，恢复原文为普通文字。

**触发方式**：右键点击锚点文字 → 上下文菜单中选择「解除追问标记」

解除后：

- 原文恢复为普通文字，下划线和数字徽章消失
- 该位置不再响应点击打开 Popover
- 追问对话数据仍保留在数据库中（不删除 Thread 和 Message），仅断开 Quote 与文字的可视化关联

### 批量管理

当一条 AI 消息上有多个锚点时，消息底部出现一个可折叠的锚点索引栏。展开后列出所有锚点的引用文字和追问轮数，支持：

- 点击跳转到对应锚点位置
- 单个解除追问链接
- 全部清除

## 七、移动端适配

### 选区触发

- 使用系统原生的长按选择代替鼠标拖选
- 选区确定后在选区上方（避让系统菜单）显示追问按钮

### Popover 形态

- 移动端 Popover 采用底部滑入的 bottom sheet 形态，而非紧贴选区
- 支持手势下滑关闭

### 输入体验

- 虚拟键盘弹出时 Popover 自动调整高度，确保输入框始终可见
- 对话区域自动收缩以腾出键盘空间

## 八、数据结构映射

UI 交互与 core-logic.md 中的数据结构对应关系：

| UI 概念         | 数据实体                  | 说明                                           |
| --------------- | ------------------------- | ---------------------------------------------- |
| 选中的文字       | `Quote.quoteText`         | 用户拖选的具体文字                              |
| 追问 Popover    | `Thread`                  | 每次划词追问创建一个子 Thread                    |
| Popover 内的对话 | `Message[]`               | Thread 内的消息列表                              |
| 认知锚点         | `Quote`                   | Quote 记录源消息、选区文字和子 Thread 的关联      |
| 嵌套追问         | `Thread.parentThreadId`   | 子 Thread 指向父 Thread                         |
| 解除锚点         | 软删除 Quote 的可视化关联  | Quote 数据保留，仅断开 UI 层的绑定               |

## 九、设计决策备忘

| 决策         | 选择             | 理由                                        |
| ------------ | ---------------- | ------------------------------------------- |
| 交互方案     | Inline Popover   | 最符合"足够轻、足够近、足够稳"原则             |
| 触发方式     | 选区 + 追问按钮   | 单一入口，简洁直觉                             |
| 同时打开数量  | 1 个             | 保持界面简洁，减少认知负担                      |
| 多轮折叠阈值  | 2 轮             | 平衡信息密度和可用空间                          |
| 嵌套视觉     | 替换 + 面包屑     | 避免 Popover 套 Popover 的视觉混乱             |
| 锚点默认态   | 淡色虚线下划线    | 极其克制，不干扰阅读                            |
| 锚点产生条件  | 至少发送 1 条消息  | 避免空锚点污染正文                              |
| 解除锚点     | 右键菜单          | 低频操作，不占用常规交互空间                     |
| 移动端形态   | Bottom sheet     | 移动端 Popover 定位不稳定，底部面板更可控        |
